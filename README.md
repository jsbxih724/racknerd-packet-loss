## 降低 RackNerd 丢包率的 5 个实操步骤

知道了丢包原因，接下来才是重点——怎么把它压下去。我自己踩过几个坑，整理成下面这套流程，按顺序做基本能压到可用状态。

**步骤一：买之前先测机房 IP**

RackNerd 官网每个机房都有 Looking Glass，能拿到测试 IP。下单前先去对应机房的 Looking Glass 页面拿到 IP，用国内的 ping 工具测一下三网延迟和丢包，把多个机房都测一遍再决定。这一步花十分钟，能帮你避开 80% 的雷。

**步骤二：买完 72 小时内一定要测 IP**

RackNerd 有个很实在的政策：购买后 72 小时内可以自助免费换 IP 一次。所以机器一拿到手，立刻用 mtr 或 ping 工具测一下三网表现，发现丢包异常（比如 IP 被某种路由策略绕路）就赶紧进后台换。

操作路径：登录客户中心 → My Services → 选中 VPS → Change IP → Proceed → 等工单确认新 IP。整个流程几分钟搞定。72 小时之后换 IP 就要收费了（一般是 3 美元左右），所以这步别拖。

**步骤三：用 mtr 而不是 ping 来定位丢包**

ping 只看端到端，mtr 能看每一跳的丢包情况。在服务器上跑 `mtr -n -r -c 100 你的国内IP`，看丢包集中在哪一跳。如果丢包发生在国内出口那一跳（通常是 163 骨干的上海/广州节点），那基本无解，是运营商的国际段拥塞；如果丢包发生在 RackNerd 上游那一跳，可以开工单让客服看一下能不能调路由。

**步骤四：上 BBR + 代理中转缓解晚高峰**

单纯靠 RackNerd 直连扛晚高峰不现实。两个常见做法：

- 服务器开启 BBR 拥塞控制（`echo "net.ipv4.tcp_congestion_control=bbr" >> /etc/sysctl.conf` 然后 `sysctl -p`），对丢包链路上的 TCP 吞吐有明显改善
- 如果是跑代理用，在国内中转节点和 RackNerd 之间走一条相对干净的线路（比如联通 4837 中转），把丢包链路绕开

**步骤五：实在扛不住就换机房或开工单**

RackNerd 支持付费迁移机房（一般几十美元），如果你最初选的机房体验实在太差，联系 Billing 工单问能不能迁到洛杉矶 DC-02 或者圣何塞。客服响应挺快，工单平均几分钟到半小时就有回复。
